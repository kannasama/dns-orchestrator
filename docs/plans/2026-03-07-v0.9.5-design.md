# v0.9.5 Design Document

**Date:** 2026-03-07
**Scope:** Bug fixes, UX improvements, dashboard enhancements, user/group/API key management

## 1. Bug Fixes

### 1.1 Views Page — Providers Column Showing "None"

**Problem:** The `ViewsView.vue` iterates over `data.provider_ids` to render provider Tag badges,
but the API response does not include this field (or returns it empty), causing all views to
display "None" regardless of attached providers.

**Fix:** Ensure `GET /api/v1/views` and `GET /api/v1/views/{id}` return the `provider_ids` array
by joining against the `view_providers` junction table. The UI rendering logic already handles
the field correctly — this is a backend-only fix in `ViewRepository` or `ViewRoutes`.

---

### 1.2 Provider Trailing Slash Breaking API Calls

**Problem:** All three providers (PowerDNS, Cloudflare, DigitalOcean) construct request paths by
appending to the base endpoint (e.g., `/client/v4/zones?name=...`). If the endpoint has a
trailing slash, this produces double slashes (`https://api.cloudflare.com//client/v4/...`) which
breaks the request.

**Fix (backend):** Strip trailing slashes from the endpoint in each provider's constructor before
storing it:

```cpp
while (!sEndpoint.empty() && sEndpoint.back() == '/') sEndpoint.pop_back();
```

**Fix (UI):** Add helper text below the API Endpoint field in `ProvidersView.vue`:
"Do not include a trailing slash."

---

### 1.3 SOA Record in PowerDNS Import

**Problem:** `PowerDnsProvider::listRecords()` returns all rrsets including SOA. When importing
records via `GET /zones/{id}/provider-records`, SOA records appear in the import list even when
`manage_soa` is false for the zone.

**Fix:** Filter at the API route level (not in the provider — providers should always return the
full truth). In the `provider-records` endpoint handler, after retrieving records from the
provider, filter out SOA records when `zone.bManageSoa == false` and NS records when
`zone.bManageNs == false`.

## 2. UX Improvements

### 2.1 Record Sorting — Group by Type, Then Name

**Current:** `ZoneDetailView.vue` sorts records by `name` only (`sortField="name"`,
`sortOrder="1"`).

**Change:** Switch to multi-column sort using PrimeVue DataTable's `multiSortMeta` prop with
`sortMode="multiple"`. Primary sort on `type` (alphabetical), secondary on `name`. Records
cluster by type (all A records together, all CNAME together) while remaining alphabetical
within each group.

---

### 2.2 Color Panel — Sakai-Style Grid

**Current:** Six accent color options (indigo, blue, teal, green, amber, rose) rendered as
swatches.

**Change:** Expand to 16 colors matching the PrimeVue Sakai demo layout. Render as two rows
of 8 circular swatches. Active selection indicated with a check mark or ring.

**Colors (two rows):**

| Row | Colors |
|-----|--------|
| 1 | noir, emerald, green, lime, orange, amber, yellow, cyan |
| 2 | sky, blue, indigo, violet, purple, fuchsia, pink, rose |

**Type change:** `AccentColor` union type in `theme.ts` expands from 6 to 16 values. The
`applyAccent()` function already uses PrimeVue's `updatePreset()` API, so it just needs the
additional palette mappings. For "noir", map primary colors to the neutral/zinc palette.

---

### 2.3 Dashboard — View Column on Zone List

**Current:** Dashboard zone table shows only the Name column.

**Change:** Add a "View" column displaying the view name associated with each zone. Use the
same view name lookup pattern from `ZonesView.vue` (fetch views list, map `view_id` to name).

---

### 2.4 Zone View Assignment — Editable on Update

**Current:** The View select field in `ZonesView.vue` is hidden during edit
(`v-if="!editingId"` on line 188). Once a zone is created with a view, it cannot be changed.

**Change (zone side):** Remove the `v-if="!editingId"` guard so the View select is visible on
both create and edit. The `PUT /api/v1/zones/{id}` endpoint already accepts `view_id` — verify
it processes updates correctly.

**Change (view side):** Add a read-only "Zones" section to the View edit drawer in
`ViewsView.vue`, showing which zones belong to the view as a list with navigation links. This
is informational only — reassignment happens from the zone side to avoid conflicting edit paths.

### 2.5 Deployment Retention — Show Default Value

**Current:** The Deployment Retention field in the zone form has no indication of what happens
when left blank.

**Change:** Add helper text below the field: "Leave blank to use system default (10)." Use
PrimeVue's `small` element or `p-text-secondary` styling beneath the InputNumber.

### 2.6 Variable Search in Record Editing

Two mechanisms added to the record edit dialog in `ZoneDetailView.vue`:

**Autocomplete on `{{`:** When the user types `{{` in the Value Template field, fetch variables
scoped to the zone via `GET /api/v1/variables?zone_id={id}` and display a filtered dropdown.
As the user continues typing, filter the list. Selecting an item inserts `{{variable_name}}`
and closes the dropdown. Standard keyboard navigation (arrow keys, Enter to select, Escape
to dismiss).

**Browse button:** A small search icon button next to the Value Template field. Clicking opens
a Popover/OverlayPanel with a searchable list of available variables showing name, scope, and
current value. Selecting a variable inserts `{{variable_name}}` at the cursor position in the
value field.

Both mechanisms use the existing `GET /api/v1/variables` endpoint with appropriate scope/zone
filtering.

### 2.7 Dashboard — Provider Health Status

**New section** on the dashboard below the stat cards.

**Backend:** Add `GET /api/v1/providers/health` endpoint. Tests each configured provider's
connectivity in parallel via `ThreadPool` using the existing `IProvider::testConnectivity()`
method. Returns an array of `{ id, name, type, status, message }` where status is
`healthy | unreachable | error`.

**UI:** Compact card or table in the dashboard: Provider Name, Type (tag badge), Status
(green/red/amber badge with label). Fetched on dashboard load — no auto-refresh.

---

### 2.8 Dashboard — Zone Sync Status with Background Refresh

**Zone table columns (expanded):** Name, View, Last Deployed (relative timestamp), Sync Status
(badge), Last Checked (relative timestamp), Actions (refresh button).

**Sync status values:**
- `unknown` (gray) — never checked
- `in_sync` (green) — desired state matches live provider state
- `drift` (amber) — differences detected
- `error` (red) — check failed (provider unreachable, etc.)

**Schema change:**

```sql
-- v005 migration
ALTER TABLE zones ADD COLUMN sync_status VARCHAR(16) NOT NULL DEFAULT 'unknown';
ALTER TABLE zones ADD COLUMN sync_checked_at TIMESTAMP WITH TIME ZONE;
```

**Status updated on three triggers:**

1. **User action:** Any preview, diff, or deployment push writes the resulting sync status and
   `now()` to the zone row.
2. **Background maintenance:** New `SyncCheckTask` registered in `MaintenanceScheduler`.
   Configurable interval via `DNS_SYNC_CHECK_INTERVAL` environment variable (default: 3600
   seconds / 1 hour). Iterates all zones, runs a lightweight diff per zone via `DiffEngine`,
   updates status. Uses `ThreadPool` for parallelism.
3. **Manual refresh:** Per-zone refresh button triggers `POST /api/v1/zones/{id}/sync-check`.
   "Refresh All" button in the dashboard section header triggers
   `POST /api/v1/zones/sync-check` (bulk). Both return updated status.

**Dashboard UX:** Refresh icon button per zone row plus a "Refresh All" button in the section
header. Shows a spinner on affected row(s) during check. "Last Checked" column uses relative
time (e.g., "12 min ago") so users can judge staleness at a glance.

## 3. User, Group & API Key Management

### 3.1 API Routes

**User routes** (`/api/v1/users`, admin-only):

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/users` | List all users (id, username, email, auth_method, is_active, groups) |
| POST | `/api/v1/users` | Create user (username, email, password, group_ids, force_password_change) |
| GET | `/api/v1/users/{id}` | Get user detail |
| PUT | `/api/v1/users/{id}` | Update user (email, is_active, group_ids) |
| DELETE | `/api/v1/users/{id}` | Deactivate user (soft delete — set is_active = false) |
| POST | `/api/v1/users/{id}/reset-password` | Admin sets new password; sets force_password_change = true |

**Group routes** (`/api/v1/groups`, admin-only):

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/groups` | List all groups (id, name, role, description, member count) |
| POST | `/api/v1/groups` | Create group (name, role, description) |
| GET | `/api/v1/groups/{id}` | Group detail with member list |
| PUT | `/api/v1/groups/{id}` | Update group (name, role, description) |
| DELETE | `/api/v1/groups/{id}` | Delete group (fails if it's a user's only group) |

**Profile routes** (authenticated user, self-service):

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/auth/me` | Already exists — add `force_password_change` to response |
| PUT | `/api/v1/auth/profile` | Update own email/display preferences |
| POST | `/api/v1/auth/change-password` | Change own password (requires current password); clears force_password_change |

**API key routes** (`/api/v1/api-keys`):

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/api-keys` | List own keys (admins see all). Returns id, label, prefix, created_at, expires_at, last_used_at. Never returns full key. |
| POST | `/api/v1/api-keys` | Create key (label, optional expiry). Returns full key **once** in response. |
| DELETE | `/api/v1/api-keys/{id}` | Revoke key (schedules deletion via existing ApiKeyRepository::scheduleDelete) |

### 3.3 Repository Changes

**UserRepository** — add methods:
- `listAll()` — returns all users with group membership
- `update(iUserId, sEmail, bIsActive)` — update user fields
- `deactivate(iUserId)` — set `is_active = false`
- `updatePassword(iUserId, sPasswordHash)` — set new password hash
- `setForcePasswordChange(iUserId, bForce)` — toggle the flag
- `addToGroup(iUserId, iGroupId)` — insert into `group_members`
- `removeFromGroup(iUserId, iGroupId)` — delete from `group_members`
- `listGroupsForUser(iUserId)` — returns group rows for a user

**GroupRepository** — new repository:
- `listAll()` — returns all groups with member count
- `findById(iGroupId)` — returns group detail with member list
- `create(sName, sRole, sDescription)` — insert group
- `update(iGroupId, sName, sRole, sDescription)` — update group
- `deleteGroup(iGroupId)` — delete group (fails if it's any user's only group)
- `listMembers(iGroupId)` — returns user rows in group

**ApiKeyRepository** — add methods:
- `listByUser(iUserId)` — returns keys owned by user
- `listAll()` — returns all keys (admin use)

### 3.4 UI Views

**UsersView.vue** — admin-only page, accessible via sidebar.
- DataTable columns: Username (monospace), Email, Auth Method (tag badge), Active (boolean
  badge), Groups (tag list), Actions (edit/delete).
- Drawer form for create: username, email, password, confirm password, group multi-select,
  force password change toggle (default: on).
- Drawer form for edit: email, active toggle, group multi-select.
- Separate action button for admin password reset.

**GroupsView.vue** — admin-only page, accessible via sidebar.
- DataTable columns: Name, Role (tag badge: admin/operator/viewer), Description, Members
  (count), Actions (edit/delete).
- Drawer form for create/edit: name, role select, description textarea.
- Expandable row or drill-down showing member list with links to user detail.

**ProfileView.vue** — accessible to all authenticated users via top-right user menu dropdown.
- Displays: username (read-only), email (editable), auth method (read-only), role (read-only).
- Change password section: current password, new password, confirm new password fields.
- Save button updates profile and/or password independently.

**ApiKeysView.vue** — accessible as a tab/section within ProfileView or as standalone page.
- DataTable columns: Label, Key Prefix (monospace, e.g., `mdn_a1b2...`), Created (relative),
  Expires (date or "Never"), Last Used (relative or "Never"), Actions (revoke button).
- Create button opens dialog: label input, optional expiry date picker.
- On successful creation, displays the full key in a copyable read-only field with a warning:
  "This key will not be shown again. Copy it now."
- Revoke uses confirm dialog before deletion.

### 3.5 Forced Password Change Flow

- `POST /api/v1/users` accepts `force_password_change` (default: `true` for new users).
- `POST /api/v1/users/{id}/reset-password` always sets `force_password_change = true`.
- `GET /api/v1/auth/me` includes `force_password_change` in response.
- **UI flow:** After login, if `force_password_change` is `true`, the router redirects to a
  dedicated password change screen (not the full profile page). The user must set a new password
  before accessing any other part of the application. After successful change, the flag is
  cleared via `POST /api/v1/auth/change-password` and the user proceeds to the dashboard.

### 3.6 Deferred: Self-Service Password Reset

Self-service password reset (forgot password flow) is planned but **not implemented in v0.9.5**.
It will require:

- SMTP/email configuration (`DNS_SMTP_HOST`, `DNS_SMTP_PORT`, `DNS_SMTP_FROM`, etc.)
- Time-limited reset token mechanism
- Potential schema additions: `reset_token` and `reset_token_expires_at` on `users` table
- A public-facing "Forgot Password" page that sends a reset email

This is deferred to a future release when email infrastructure support is added.
